§1 Justifications architecturales

FIX-05 — MerciPage cohérence invoice_number + D48 SELLER PASS

Alternatives considérées

  1. A — Rendre invoice_number côté serveur et l'inclure comme prop dans un nouveau routeur SSR. Rejeté : refactor SPA lourd, hors scope v3.4.6.
  2. B — Lire invoice_number depuis sessionStorage avec fallback génération client (priorité URL > sessionStorage > fallback). Retenu.
  3. C — Stocker dans localStorage (persistance cross-session). Rejeté : violation RGPD si client partage son navigateur, et MerciPage n'est atteint qu'une fois par commande.

Choix retenu — justification technique

Cascade de priorité URL query > sessionStorage > génération client :

  • URL query (?invoice=AEGIS-...) = cas du lien email reçu par le client (CHANGE-07).
  • sessionStorage (aegis_diag_request.invoice_number) = cas du retour post-paiement Mollie sans paramètre URL (flow nominal).
  • Fallback generateInvoiceNumber() = edge case où ni URL ni sessionStorage n'existent (client ayant désactivé stockage, ou navigation directe). Le client voit alors un identifiant plausible même en dégradé, ce qui évite un crash UI.

Référence standards

RFC 6265 sessionStorage semantics (persistence limited to tab lifetime). Cohérence avec l'invariant FIX-04 bridge T0315 (un seul identifiant client-facing).

Trade-offs acceptés

DETTE21 (refactor DRY src/config/seller.ts partagé) reportée v3.4.7 — duplication SELLER MerciPage/email-templates acceptable si les deux copies sont strictement alignées (vérifié grep).

FIX-06 — Template Pandoc/XeLaTeX de référence PASS

Alternatives considérées

  1. A — Modifier le template de production existant dans C:\Users\jpcha\Downloads\diagnostics\. Rejeté : hors périmètre MCP confirmé JP T1023.
  2. B — Produire un template de référence minimal D48 + placeholders signature dans le périmètre MCP. Retenu T1023 JP.
  3. C — Ne rien livrer et documenter JP_INPUT_NEEDED. Rejeté : bloquerait la mission, JP a explicitement levé ce blocage T1023.

Choix retenu — justification technique

Template scripts/diagnostic-template-reference.tex auto-documenté, conforme D48 (inputenc utf8 + fontenc T1 + babel french), avec variables Pandoc natives ($invoice$, $approved_by$, $approved_at$, $pdf_sha256$) et bloc signature eIDAS Art. 25 + footer SELLER cohérent MerciPage/emails.

Inclusion d'une conditionnelle Pandoc $if(pdf_sha256)$...$endif$ pour supporter deux modes : pipeline mono-pass (sans empreinte) et double-pass (avec hash injecté post-génération).

Référence standards

  • eIDAS Article 25 (Règlement UE 910/2014) — signature électronique simple (SES) recevable en preuve devant les juridictions EU.
  • RFC 4648 §8 — encodage SHA-256 hexadécimal (64 caractères).
  • Pandoc Template Syntax — $var$ + $if(var)$...$endif$ (pandoc.org/MANUAL.html#templates).

Trade-offs acceptés

Le template est une référence, pas le template de production. Merge manuel JP requis post-mission : copier les blocs D48 (packages + SELLER) + bloc signature vers C:\Users\jpcha\Downloads\diagnostics\AEGIS-DIAGNOSTIC-template.tex. Documentation merge en fin de fichier .tex.

FIX-07 — Script PowerShell pipeline PDF PASS

Alternatives considérées

  1. A — Pipeline pandoc mono-pass avec hash pré-calculé sur le markdown (pas le PDF). Rejeté : l'empreinte doit certifier l'intégrité du PDF final livré au client, pas de la source.
  2. B — Pipeline double-pass : 1ère passe génère PDF intermédiaire → SHA-256 calculé → 2ème passe ré-injecte le hash dans le footer signature. Retenu.
  3. C — Post-process avec pdftk ou qpdf pour injecter métadonnées hash sans re-render. Rejeté : ajoute une dépendance externe et ne garantit pas la présence visible du hash dans le footer rendu.

Choix retenu — justification technique

Script PowerShell avec 7 étapes documentées dans l'en-tête : (1) validation markdown ; (2) validation paramètres (regex ^AEGIS-\d{8}-\d{4}$ + sanitize LaTeX [$\{}] sur approved_by) ; (3) Pandoc passe 1 → PDF intermédiaire ; (4) Get-FileHash -Algorithm SHA256 ; (5) Pandoc passe 2 avec --variable=pdf_sha256:$hash ; (6) log JSONL audit append-only ; (7) retour PSCustomObject.

Résolution template automatique : priorité template production JP (C:\Users\jpcha\Downloads\diagnostics\AEGIS-DIAGNOSTIC-template.tex) si présent, sinon template référence FIX-06.

Support -DryRun pour valider les paramètres sans invocation Pandoc (utile CI/tests).

Sécurité R_T1155_01

  • Aucun credential en log (valeurs $approved_by sanitized, jamais $env:*).
  • Chemins sensibles relativisés via Get-RelativePath (masque C:\Users\jpcha\... → ~/...).
  • Validation stricte inputs (regex invoice, trim + range length approved_by, rejet caractères LaTeX dangereux).
  • Logs JSONL append-only dans C:\Projects\aegis-ops\logs\aegis-delivery-audit-YYYYMMDD.jsonl.

Trade-offs acceptés

Comptage pages PDF heuristique via parsing binaire /Type /Page (approximatif). Alternative cleaner pdfinfo (poppler) reportée si besoin d'exactitude. Suffisant pour audit trail.

FIX-08 — Signature numérique eIDAS Art. 25 SES PASS

Alternatives considérées

  1. A — Signature qualifiée eIDAS (QES) via prestataire agréé (DocuSign, Universign). Rejeté v3.4.6 (DETTE17) : coût récurrent + intégration tierce non justifiée pour DIAGNOSTIC 250 €.
  2. B — Signature électronique simple (SES) eIDAS Art. 25 : nom + timestamp + hash SHA-256 du PDF. Défensible en preuve devant juridictions EU pour B2B valeur modérée. Retenu.
  3. C — Pas de signature (MVP). Rejeté : exigence CGI (Compliance Gouvernance Intégrée) introduite par JP T0905 incompatible avec MVP sans signature.

Choix retenu — justification technique

Trois ajouts coordonnés :

  • Interface MailerPaymentData étendue de 4 champs optionnels (approved_by, approved_at, pdf_sha256, signature_note) — rétro-compatible avec les envois pré-FIX-08.
  • Email template deliveryConfirmationHtml : bloc signature conditionnel ${data.approved_by ? ... : ''} — n'apparaît que si approbateur fourni, permettant usage legacy sans signature.
  • Route POST /api/send-delivery : validation stricte approved_by requis (2-100 chars), approved_at ISO 8601 parse (server default new Date().toISOString()), pdf_sha256 regex ^[a-f0-9]{64}$, signature_note sanitize HTML + slice 200 chars. Log audit JSONL delivery_signed avant envoi (R_T1155_01).

Référence standards

  • Règlement (UE) n° 910/2014 (eIDAS) Article 25 §1 : recevabilité de la signature électronique simple.
  • NIST FIPS 180-4 — SHA-256 specification.
  • OWASP ASVS V5 — input validation server-side (regex + range checks).

Trade-offs acceptés

  • DETTE17 — QES v3.5 si engagement juridique > 5000 € nécessite preuve qualifiée.
  • DETTE20 — Hashchain blockchain audit reportée : log JSONL append-only suffit MVP.
  • Signature visible uniquement dans email client + footer PDF (via FIX-06 template). Audit trail JSONL côté serveur JP.
