Préambule causal

À l'ouverture de la session 20260507T0935 (post MAJ Desktop v1.6259.1), JP a soulevé une question légitime restée jusqu'alors implicite : pourquoi l'annonce textuelle du system prompt Anthropic indique-t-elle « web/mobile chat interface » alors que l'environnement réel est manifestement ACDC Desktop avec MCP Filesystem actif ? Cette section formalise la distinction et son traitement opérationnel.

L'environnement Claude perçu par AEGIS résulte de la superposition de trois couches distinctes. La confusion vient du fait que la couche cœur annonce textuellement un environnement, alors que la surcouche client en modifie la réalité fonctionnelle sans modifier l'annonce textuelle. Sans protocole de vérification empirique, Claude croit l'annonce et calibre ses outils en conséquence — risque de mauvaise sélection d'outils ou d'affirmations erronées sur les capacités disponibles.

Anatomie en trois couches

Le diagramme ci-dessous représente la stratification réelle de l'environnement. Chaque couche a une source distincte, une autorité distincte, et une visibilité distincte pour Claude. La discordance entre la couche 1 (textuelle) et la couche 3 (empirique) est le cœur du problème.

Couche 01 — Cœur Anthropic

System Prompt Anthropic

Source : envoyé par Anthropic à chaque session — identique tous clients (web, mobile, Desktop)
Texte cœur définissant l'identité Claude, ses règles de comportement, son cadre de sécurité. Mentionne explicitement l'interface attendue. « Claude is currently operating in a web or mobile chat interface run by Anthropic. » Autorité : maximale (Anthropic). Visibilité : intégrale (Claude le lit en clair). Mutabilité : aucune côté utilisateur.
superposition silencieuse
Couche 02 — Surcouche client

Wrapper Desktop (Electron + CC embarqué)

Source : application Claude for Windows v1.6259.1 — distincte selon le client utilisé
Surcouche logicielle qui injecte des capacités supplémentaires sans modifier le texte du system prompt cœur. Active des outils MCP, des onglets (Chat, Cowork, Code), des tools deferred chargés à la demande via tool_search. Le binaire Desktop évolue à son propre rythme (cycle ~hebdomadaire) indépendamment du modèle.

Autorité : intermédiaire (Anthropic, mais distribuée par binaire client). Visibilité : opaque pour Claude (pas de texte explicite sur la surcouche). Mutabilité : MAJ automatique du binaire.
manifestation empirique
Couche 03 — Réalité fonctionnelle

Environnement empirique observable

Source : preuve directe par appel d'outil — seule source de vérité opérationnelle
Capacités réellement disponibles dans la session : MCP Filesystem connecté, répertoires autorisés, outils deferred découvrables via tool_search, intégrations Chrome, Google Drive, Gmail, Calendar, Gamma. tool_search('filesystem') → 5 outils chargés
list_allowed_directories → C:\Projects, C:\Users\jpcha\.antigravity, Downloads, C:\Projects\Documents Autorité : empirique absolue (preuve d'exécution). Visibilité : conditionnelle (Claude doit appeler l'outil pour la révéler). Signature : 4 dirs = ACDC Desktop confirmé.
Discordance cœur ↔ réalité

La couche 1 dit « web ». La couche 3 prouve « Desktop ». Que croire ?

  • Sans Gate D206 — Claude croit la Couche 1 (annonce textuelle), suppose absence d'outils MCP, refuse d'écrire des fichiers locaux, propose à JP d'utiliser des téléchargements via create_file. Friction inutile, perte de capacités réelles.
  • Avec Gate D206 — Claude appelle tool_search('filesystem') + list_allowed_directories, observe la signature ACDC Desktop, bascule mentalement sur Couche 3 comme source de vérité, déploie les bons outils.
  • Cause racine — l'annonce du system prompt cœur ne peut pas être mise à jour dynamiquement par le wrapper client. C'est une limitation architecturale, pas un bug. La résolution est protocolaire (vérification empirique), pas technique.

Gate D206 — Résolution causale Pearl

La Gate D206 (mémoire #30) formalise le protocole de vérification empirique exclusive (VEE v1.1, 24/04/2026). Elle est exprimée selon les trois niveaux de Judea Pearl : Observation, Intervention, Contrefactuel — alignée avec la règle absolue L_T1230_01 (mémoire #26).

N1 — OBSERVATION

Constat de la discordance

Que voit-on quand on lit le system prompt ?

L'annonce textuelle déclare un environnement (« web/mobile »). En parallèle, le contexte de session contient des éléments suggérant une surcouche : références à des outils non chargés par défaut, présence de fichiers projet en /mnt/project/, mentions de filesystem MCP dans les memories. Signal d'incohérence déclenché.

Wrapper announce ≠ runtime reality
N2 — INTERVENTION

Action de vérification empirique

Qu'arrive-t-il si on agit pour observer ?

Claude exécute tool_search('filesystem'), puis list_allowed_directories. Le retour est binaire : soit aucun outil filesystem n'est disponible (claude.ai web/mobile pur), soit 4 répertoires sont autorisés (ACDC Desktop confirmé). L'intervention bascule la source de vérité de la Couche 1 vers la Couche 3.

tool_search → list_allowed_directories → signature
N3 — CONTREFACTUEL

Que se serait-il passé sans D206ㅤ?

Et si le protocole n'avait pas été appliqué ?

Sans Gate D206, Claude répond depuis la Couche 1 et propose des contournements inutiles (téléchargements, copier-coller manuel, recommandations basées sur l'hypothèse d'absence d'outils). Le coût est triple : token (réponse plus longue), friction utilisateur (étapes manuelles), risque d'erreur (fausses affirmations sur les capacités).

Pas de D206 → réponses sub-optimales transversales
